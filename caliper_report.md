# 結合テスト自動化 活動報告

# 1. 背景・課題

本プロジェクトでは、Hyperledger Fabric を用いた分散型ブロックチェーンサーバおよび Web API サーバを開発している。

ウォーターフォール型開発のため、設計書・テスト仕様書・テスト手順書などのドキュメントは Excel ベースで管理されており、ドキュメント作成やテスト実行を手動で実施していた。

特に結合テストでは、

- 複数サーバを跨ぐ試験
- ノード停止などの非定常試験
- データ改ざん試験
- 再試験（リグレッション）

が多数発生するため、以下の課題があった。

- テスト工数が非常に大きい
- 不具合修正のたびに再試験が必要
- 手順ミスによるやり直しが発生
- 試験品質が担当者スキルに依存
- ドキュメント作成負荷が大きい

そのため、結合テスト工程全体の効率化・標準化が必要となった。

---

# 2. 単体テストと結合テストの違い

| 項目 | 単体テスト | 結合テスト |
|---|---|---|
| 対象 | 単一機能 | 複数システム連携 |
| 確認内容 | ロジック正当性 | システム間連携 |
| 環境依存 | 少ない | 非常に大きい |
| テストデータ | 単純 | 複雑 |
| 実行速度 | 高速 | 比較的低速 |
| 自動化難易度 | 比較的容易 | 非常に高い |

## なぜ単体テストは自動化しやすいのか

単体テストは、1つの関数やクラス単位で動作確認を行うため、

- 必要なデータが少ない
- 外部システム依存が少ない
- 実行条件が単純
- テスト環境を局所化できる

という特徴があり、自動化しやすい。

## なぜ結合テストは自動化が難しいのか

一方、結合テストでは、

- 複数サーバの連携
- DB状態
- ネットワーク状態
- ノード停止
- 非同期処理
- テストデータ準備
- 試験順序依存

など、多数の前提条件を揃える必要がある。

特に Hyperledger Fabric では、

- ノード停止時の挙動
- 分散台帳整合性
- データ改ざん検知
- 重複登録防止

など、ブロックチェーン特有の観点が必要となり、自動化難易度が高かった。

---

# 3. 今回の取り組み

## 実施内容

単なるテスト実行自動化ではなく、結合テスト工程全体のワークフロー自動化基盤を構築した。

具体的には、

- 設計書からテスト手順書を自動生成
- テスト手順書からテストコードを生成
- テスト実行結果を Excel 出力
- 試験証跡を自動整形

を実現した。

これにより、従来手作業で実施していた結合テストプロセスを大幅に効率化した。

---

# 4. 工夫点

## 工夫点① 複雑な事前準備をスクリプト化

結合テストでは、

- 改ざんデータ作成
- HLFノード停止
- 非定常状態生成

など、事前準備が複雑だった。

この課題に対し、

- データ改ざんスクリプト
- ノード停止スクリプト
- テストデータ生成スクリプト

を事前に部品化した。

その結果、

「複雑な試験条件」を「スクリプト呼び出し」に変換でき、  
テストコード自動生成の難易度を大幅に低減できた。

---

## 工夫点② テスト観点テンプレートによる品質向上

単純な自動生成では、テスト観点が不足し、試験品質にばらつきがあった。

そこで、

- 異常系
- 非定常系
- 分散システム特有観点
- ブロックチェーン特有観点

をテンプレート化し、テストケース生成時に注入する仕組みを構築した。

これにより、

- ベテラン依存の低減
- 属人化防止
- ブロックチェーン特有リスクへの対応
- テスト品質向上

を実現した。

---

# 5. 成果

- テスト手順書作成時間 50%削減
- テスト実行工数 50人日削減
- リグレッション確認高速化
- 不具合修正時の確認効率向上
- 手順ミスによるやり直しゼロ
- 属人化低減
- 非定常試験の再現性向上

---

# 6. まとめ

単体テスト自動化に加え、結合テスト工程全体を自動化することで、

- 工数削減
- 品質向上
- 属人化低減
- 再試験効率化

を実現した。

また、単純な自動化に留まらず、

- ブロックチェーン特有リスク
- 分散システム特有観点
- 非定常試験

を組み込むことで、品質面でも高い効果を得られた。

今後は、さらなる自動化範囲拡大や CI/CD 連携による継続的品質向上を推進していく。
